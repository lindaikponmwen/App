
# DrLevy.AI File Synchronization & S3 Integration

This document details the architecture and usage of the file synchronization system between the local client (IndexedDB), the Application Server (PHP/MySQL), and Object Storage (AWS S3).

## 1. System Architecture

The system utilizes a **Direct-to-S3** upload pattern using Presigned URLs. This ensures that large file transfers do not consume application server bandwidth or memory.

### Data Flow
1.  **Request URL:** The Client asks the PHP API for a temporary, signed URL to upload a specific file.
2.  **Generate Signature:** The PHP API uses AWS Credentials to generate a Signature V4 URL valid for a short time (e.g., 1 hour).
3.  **Direct Upload:** The Client uploads the file binary directly to AWS S3 using the signed URL.
4.  **Log Metadata:** Upon successful upload, the Client sends file metadata (name, size, path) to the PHP API to store in the SQL database.
5.  **Sync State:** The Client updates the local IndexedDB to mark the file as synced and stores the returned database ID (`uid`).

## 2. Database Schema

The system relies on two primary tables in the MySQL database. Run `api/setup_database.php` to initialize them.

### `user_files`
Stores metadata for all files associated with users and projects.
*   **id**: Auto-incrementing primary key.
*   **uid**: A unique 8-12 char alphanumeric string used for frontend referencing.
*   **user_id**: ID of the file owner.
*   **project_id**: ID of the project the file belongs to.
*   **file_path**: The full S3 key (e.g., `Data/study_data.csv`).
*   **active**: Boolean (1 = Active, 0 = Deleted).
*   **file_size**, **file_type**, **category**: Metadata.

### `project_sync_state`
Tracks the synchronization status of a project to detect if a client is out of date.
*   **project_id**: Primary key.
*   **sync_hash**: A generic hash generated by the client after a successful sync.
*   **last_synced_at**: Timestamp of the last successful sync.

## 3. PHP API Endpoints

### `GET /api/generate-presigned-url.php`
Generates an AWS Signature V4 URL for PUT (Upload) or DELETE operations.
*   **Parameters:** `filename`, `prefix`, `method` (PUT/DELETE).
*   **Response:** JSON containing the `url`, `key`, and `method`.

### `POST /api/log_file.php`
Records file metadata in the database after a successful S3 action.
*   **Payload:** `user_id`, `project_id`, `file_name`, `file_path`, `file_size`, `active` (1 or 0).
*   **Response:** JSON containing a success message and the `uid`.

### `GET /api/get_project_files.php`
Retrieves a list of all active files for a specific project. Used during project initialization to populate the file tree.
*   **Parameters:** `project_id`.
*   **Response:** JSON array of file records.

### `POST /api/update_sync_hash.php`
Updates the synchronization hash state for a project.
*   **Payload:** `project_id`, `sync_hash`.

## 4. Frontend Integration (`src/services/s3Service.ts`)

The frontend logic handles the synchronization lifecycle.

### Key Functions

*   **`uploadFileToS3(fileNode, path, prefix)`**:
    1.  Calls `generate-presigned-url.php` to get a PUT URL.
    2.  Performs a `fetch` request with the binary Blob to S3.
    3.  Calls `log_file.php` to record the upload.
    4.  Updates the local IndexedDB node with `is_synced: true` and the returned `db_id` (uid).

*   **`syncProject(uploads, deletions, prefix, onProgress)`**:
    1.  Iterates through a list of pending deletions first.
    2.  Iterates through a list of unsynced uploads.
    3.  Reports progress via a callback for UI feedback.

## 5. User Guide & Settings

The synchronization features are accessible via the **Settings Page** within the application.

### Manual Sync
1.  Navigate to **Settings** > **Cloud Backup**.
2.  The **Sync Status** panel displays pending changes.
    *   It lists files present locally but not marked as synced.
    *   It lists files deleted locally that need to be removed from the cloud.
3.  Click the **Sync Now** button.
4.  A progress bar will indicate the status of uploads and deletions.

### Auto-Sync
1.  Navigate to **Settings** > **Cloud Backup**.
2.  Toggle the **Auto-Sync** switch to **ON**.
3.  Adjust the **Sync Interval** slider (1 to 60 minutes).
4.  The application will automatically check for changes in the background. If unsynced files are found, it will trigger a silent sync operation without blocking the user interface.

### Project Restoration
When a user loads a project:
1.  The app checks `get_sync_state.php`.
2.  It compares the remote hash with the local storage hash.
3.  If they differ, or if local data is missing, it calls `get_project_files.php`.
4.  It reconstructs the file tree structure and fetches content from S3 on-demand using presigned GET URLs.

## 6. Configuration

To configure the backend:
1.  Edit `api/config.php`.
2.  Set `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_DEFAULT_REGION`, and `AWS_BUCKET`.
3.  Ensure `S3_FOLDER_PREFIX` is set correctly if utilizing a subfolder in the bucket.
4.  Run `api/setup_database.php` once to create the necessary tables.
